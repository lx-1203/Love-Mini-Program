## 1. 项目概述

**项目名称**：校园恋爱小程序（Campus Love Mini-Program）

**项目目标**：为在校大学生提供一个基于校园身份认证的社交恋爱平台，核心指标为**用户引流效果**和**客户停留时间**。

**目标用户**：全国在校大学生、研究生（通过学校认证验证身份，非在校生无法使用核心功能）。

**北极星指标**：用户引流效率 + 客户停留时长。

**核心约束（不可违反）**：
- 🚫 严禁游戏化元素（积分、等级、排行榜、成就徽章、虚拟货币等）
- 🚫 严禁电商/购物功能（商城、付费商品、虚拟物品交易等）
- ✅ 学校认证为硬门禁——未认证用户无法使用社交功能
- ✅ 资料完整度 100% 为解锁全部功能的硬性前提（寻觅页始终开放以引流）

## 3. 信息架构

### 3.1 一级入口（底部 TabBar — 5 个）

```
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│   寻觅    │   喜欢    │   村口    │   消息    │   我的    │
│ Discover │  Likes   │ Village  │ Messages │ Profile  │
├──────────┼──────────┼──────────┼──────────┼──────────┤
│ 始终开放  │ 资料解锁  │ 资料解锁  │ 资料解锁  │ 资料解锁  │
└──────────┴──────────┴──────────┴──────────┴──────────┘
```

**锁定规则**：资料未 100% 完善时，除「寻觅」外所有 Tab 锁定，显示模糊头像 + 引导文案 + "立即完善"按钮。

### 3.2 次要入口

- **活动 (Activities)**：从寻觅页卡片耗尽后展示，或从首页/我的页进入
- **兴趣圈 (Interest Circles)**：从我的页进入，精选话题出现在村口「兴趣圈」分类
- **每日一问 (Daily Question)**：从寻觅页/我的页进入
- **签到 (Check-in)**：从寻觅页顶部/我的页进入

## 4. 社交升温漏斗

本项目核心设计理念：将用户从"陌生人"渐进引导至"深度关系"的六层漏斗。

```
┌────────────────────────────────────────────────────────────┐
│                    社交升温漏斗                              │
├────────┬──────────────────┬────────────────────────────────┤
│ 层级    │ 功能载体           │ 社交深度                         │
├────────┼──────────────────┼────────────────────────────────┤
│ L1 曝光 │ 卡片滑动/每日推荐    │ 单向表达兴趣，无回馈压力            │
│ L2 关注 │ 喜欢/访客/超级喜欢   │ 单向到双向过渡，产生社交期待          │
│ L3 匹配 │ 心动信号/倒计时      │ 双向确认，紧迫感促进行动             │
│ L4 沟通 │ 私信/临时匿名聊天     │ 双向实时交流，建立初步信任            │
│ L5 圈子 │ 村口/兴趣圈/每日一问  │ 群体互动，扩展社交半径               │
│ L6 场景 │ 线下活动/校园场景     │ 线下见面，关系落地                  │
└────────┴──────────────────┴────────────────────────────────┘
```

## 5. 功能模块详细需求

### 5.1 寻觅（Discover）— L1~L2 曝光+关注层

| 子功能 | 交互设计 | 社交心理学原理 |
|--------|----------|----------------|
| 全屏卡片滑动 | 沉浸式单卡片展示，左滑拒绝/右滑喜欢/上滑超级喜欢 | 减少决策疲劳，降低选择焦虑 |
| 每日限量推荐 | 默认 10 张/天，中午 12 点定时刷新 | 稀缺感制造期待，定时回流动因 |
| 挽回机制 | 每日 1 次挽回最后一位拒绝的用户 | 后悔机制，延长决策链条 |
| 卡片详情 | 点击展开完整资料（照片/学校/兴趣/简介） | 信息分层，先感性后理性 |
| 底部村口动态流 | 卡片耗尽后展示热门帖子 | 无缝切换，不中断浏览流 |

**卡片内容**：头像、昵称、学校、年级、专业、兴趣标签（最多5个）、个人简介（≤200字）。

**替代入口**：卡片耗尽后展示"更多嘉宾"、"逛逛村口"、"每日一问"入口。

### 5.2 喜欢（Likes）— L2~L3 关注+匹配层

**双标签页**：
- **「喜欢我的」**：横向头像列表 + 纵向详情，展示对方头像、昵称、学校、喜欢时间
- **「访客」**：记录谁看过我，展示头像、昵称、访问时间

**双向喜欢检测**：用户 A 喜欢用户 B 时，系统自动检测 B 是否已喜欢 A。

**心动信号（Heart Signal）**：
- 双向喜欢时自动触发，双方收到推送通知
- 展示对方头像、学校、年龄、简介亮点
- **24 小时倒计时**内可"直接开聊"
- 超时后心动信号过期，但匹配关系保留

**资料未完善锁定**：显示模糊头像 + "完善资料后查看谁喜欢了你" + "立即完善"按钮。

### 5.3 村口（Village）— L5 圈子层

**三大标签切换**：关注 / 同城 / 发现

**六大分类筛选**：全部 / 兴趣圈 / 诚意帖 / 同乡 / 蒙面 / 最新

**帖子卡片**：
- 用户身份标签（学校、年级、专业）
- 图文内容（富文本，最多 500 字）
- 话题标签（#兴趣圈话题 #同城活动 等）
- 互动数据（点赞数、评论数、转发数）
- 定位信息（可选）

**帖子互动**：
- 点赞（可取消，去重）
- 评论（列表 + 发表，支持回复）
- 关注作者（一键关注，互关后显示标识）
- 私信作者（资料完善后可发）
- 转发/分享

**发帖功能**：
- 悬浮 FAB 按钮触达发帖页
- 文字输入（限制 500 字）
- 图片上传（最多 9 张）
- 话题标签选择（自定义 + 热门推荐）
- 分类选择（从六大分类中选）
- 定位添加（可选）

**帖子详情**：顶部作者信息卡片 + 图文内容 + 评论列表 + 互动按钮

### 5.4 消息（Messages）— L3~L4 匹配+沟通层

**消息分类**：
- **心动信号 Banner**（顶部）：双向匹配摘要 + 24h 倒计时 + "直接开聊"入口
- **私信会话列表**：头像、昵称、最后消息摘要、未读计数、时间
- **系统通知**：喜欢通知、访客通知、评论通知、关注通知、匹配通知

**私信详情**：
- 聊天界面：支持文字、图片消息
- 实时消息推送（WebSocket）
- 消息已读/未读状态
- 在线状态显示

**临时匿名聊天**：
- 24 小时过期，倒计时显示
- 匿名保护（不暴露真实身份信息）
- 到期前可申请联系方式交换（双向确认）
- 交换成功后暴露微信号/手机号

**系统通知**：
- 分类展示：喜欢/访客/评论/关注/匹配
- 每条通知带跳转链接（跳转到对应页面）
- 已读/未读状态
- 批量标记已读

### 5.5 我的（Profile）— 个人中心

**个人信息展示**：
- 头像、昵称、学校、个人统计（获赞数、粉丝数、关注数、访客数）
- 会员标识（如有）

**资料完善度**：
- 进度条 + 百分比
- 各字段权重：
  - 头像：20%
  - 昵称：10%
  - 性别：10%
  - 生日：10%
  - 学校：20%
  - 年级+专业：10%
  - 兴趣标签：10%
  - 个人简介：10%

**功能菜单**：
- 编辑资料（基本信息 + 头像上传）
- 学校认证（学生证上传 + 学号 + 管理员审核）
- 推荐计划设置（同校优先/同城/不限 + 时间偏好）
- 兴趣圈管理
- 签到
- 每日一问
- 反馈中心
- 设置

**学校认证流程**：
1. 用户上传学生证照片 + 填写学号
2. 管理员后台审核
3. 审核通过后获得「已认证」标识
4. 未认证用户无法使用社交功能

### 5.6 辅助功能模块

**签到 (Check-in)**：
- 每日签到，记录连续天数
- 签到后额外获得推荐机会（+3 次/天）
- 连续天数仅作视觉标识（无积分/等级！）

**每日一问 (Daily Question)**：
- 每日一个问题
- 用户可回答（支持匿名）
- 可浏览他人回答
- 共同回答作为推荐加分项

**兴趣圈 (Interest Circles)**：
- 圈子列表、加入/退出
- 圈内话题发布/回复
- 精选话题出现在村口「兴趣圈」分类

**活动 (Activities)**：
- 活动列表 + 详情 + 报名
- 活动状态：报名中 / 进行中 / 已结束 / 预告
- 展示已报名人数、时间、地点
- 报名后可查看活动日历

**反馈中心 (Feedback)**：
- 用户提交反馈工单
- 支持分类：建议 / Bug / 投诉 / 其他
- 提交后可查询处理状态

## 7. Mock/Real 双模式设计

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| Mock | 客户端使用本地 fixtures，后端启动 mock profile | 本地开发、单元测试 |
| Real | 客户端调用真实后端 API，后端连接 MySQL | 集成测试、生产环境 |

**核心要求**：Mock 与 Real 必须覆盖同一套主流程，行为一致。

## 8. 非功能需求

### 8.1 安全性
- JWT access + refresh token 认证
- Token 黑名单机制（登出时失效）
- CORS 跨域白名单
- 请求速率限制（每个用户、每个 IP）
- 敏感词过滤（帖子、评论、私信）
- 安全响应头（CSP、X-Frame-Options 等）

### 8.2 性能
- 推荐服务 N+1 查询优化
- 分页加载（默认 20 条/页）
- 图片懒加载
- WebSocket 心跳 + 重连

### 8.3 兼容性
- 微信小程序（主要目标平台）
- H5 网页端（辅助）
- 处理平台差异（API 适配层）

### 8.4 测试
- 前端：Vitest 单元测试 + 组件测试
- 后端：JUnit 5 单元测试 + MockMvc 集成测试
- Python 脚本：pytest + mypy strict + ruff
- 契约测试：OpenAPI Spectral lint
