## 四、权限管理评估

### 4.1 角色定义
- ✅ 后端 `User.java` 定义 `role` 字段（值 `ADMIN` / `USER`）
- ❌ 前端 `session.ts:61` 用小写 `"admin"` 校验，**与后端大写不一致**
- ❌ JWT token 无 `role` claim

### 4.2 权限校验机制
- ✅ 前端路由守卫 `router/index.ts` 检查登录态
- ❌ 前端无角色校验（任何登录用户都能进管理后台）
- ✅ 后端 `JwtAuthenticationFilter` 校验 token
- ❌ 后端管理接口未使用 `@PreAuthorize` 或角色拦截器
- ❌ Mock profile 下 `permitAll` 全部放行（误部署即全员可调 admin 接口）

### 4.3 角色隔离完整性
- ❌ 普通用户 token 可能访问管理接口（无角色校验）
- ❌ 管理员密码明文比较（`RealAuthService.java:213`）
- ❌ 无登录失败锁定机制
- ❌ 无验证码
- ❌ Logout 后 token 仍有效 24h（无黑名单）

## 十、测试结论

管理员视角下，管理端**仅完成登录与页面骨架**，**核心管理功能全部为硬编码 Mock**，**后端管理 API 严重缺失**（12 类核心接口未实现），**权限控制存在致命漏洞**（角色大小写不一致、Mock 全放行、密码明文）。**审计日志体系完全未实现**。

管理端**不具备生产可用性**，需在后端补齐管理 API、前端对接真实接口、修复权限校验漏洞后方可交付。
