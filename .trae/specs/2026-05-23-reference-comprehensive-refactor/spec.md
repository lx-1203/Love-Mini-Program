# 参考内容全面分析与重构方案 Spec

## Why

基于对参考目录（`D:\6\恋爱小程序\参考`）中11张竞品"青藤"微信截图的深度分析，结合对当前项目代码库的全面审查（含7个后端Real服务、35+张数据库表、前端Mock/Real双模式Store），发现项目虽经多期重构已具备较完整的基础架构，但仍存在以下核心问题：

1. **二级社交功能有表无服务**：兴趣圈/每日一问/签到已有数据库表和前端页面，但后端缺少对应的Real Service实现，Real模式下这些功能不可用
2. **推荐算法维度缺失**：RealRecommendationService缺少兴趣标签匹配加权，讨论推荐和推荐历史返回空列表
3. **前端Store Real模式未验证**：9个Store的Real模式API调用链路未经完整验证，profile.ts甚至缺少Mock模式支持
4. **数据模型不一致**：heart\_signals表缺少match\_type列，帖子点赞无去重机制，临时聊天表无Real服务
5. **设计预览与实际实现偏差**：设计预览的首页/TabBar/匹配页与实际pages.json路由结构不一致

本次方案核心目标：**从"基础架构完整+二级功能缺失"升级为"全功能真实驱动+完整社交闭环"**，精准还原参考产品的核心功能体验，保留大学生模式的独特性，集成在线社交互动功能，确保推荐方案设置完整可用。

**特别约束**：严禁任何游戏化元素（积分/等级/排行榜/成就徽章等）和购物相关功能（商城/付费/虚拟物品等）。项目北极星指标：**提升用户引流效果 + 延长客户停留时间**。

### 二、交互流程分析

#### 核心用户旅程（5阶段闭环）

```
阶段1: 引流
  微信登录 → 学校认证 → 资料完善 → 解锁全部功能

阶段2: 发现
  寻觅页浏览推荐卡片 → 左滑拒绝/右滑喜欢

阶段3: 匹配
  双向喜欢 → 触发心动信号 → 倒计时内开聊

阶段4: 互动
  私信对话 → 村口社区互动 → 兴趣圈话题 → 活动报名

阶段5: 留存
  每日推荐刷新 → 签到 → 每日一问 → 消息通知 → 社区动态更新
```

#### 关键交互细节

1. **卡片滑动**：全屏沉浸式，手势流畅，飞出动画自然
2. **心动信号**：双向喜欢后自动触发，24h倒计时制造紧迫感
3. **村口发帖**：悬浮FAB按钮，随时可发，降低创作门槛
4. **私信入口**：多个触点（喜欢页/帖子详情/心动信号/兴趣圈），缩短社交距离
5. **时间门控**：每日中午12点刷新推荐，制造期待感和回流动力
6. **多层级社交**：轻量互动（滑动/点赞）→ 中度互动（关注/话题）→ 深度互动（私信/活动）

## ADDED Requirements

### Requirement: RealCircleService兴趣圈完整实现

系统 SHALL 实现兴趣圈的完整后端Real服务，替代Mock服务，使兴趣圈功能在Real模式下可用。

#### Scenario: 获取兴趣圈列表

* **WHEN** 用户访问兴趣圈页面

* **THEN** 系统从数据库查询所有兴趣圈

* **AND** 每个圈展示名称、图标、成员数量、最新话题摘要

* **AND** 标记当前用户是否已加入

#### Scenario: 加入/退出兴趣圈

* **WHEN** 用户点击"加入"按钮

* **THEN** 系统创建CircleMembership记录

* **AND** 圈子成员数+1

* **AND** 用户可在"我的兴趣圈"中看到

#### Scenario: 发布圈内话题

* **WHEN** 用户在兴趣圈内发布话题（标题+正文+可选图片）

* **THEN** 系统创建CircleTopic记录

* **AND** 话题出现在圈内话题列表

* **AND** 精选话题可出现在村口"兴趣圈"分类

#### Scenario: 回复圈内话题

* **WHEN** 用户回复兴趣圈话题

* **THEN** 系统创建CircleReply记录

* **AND** 话题回复数+1

* **AND** 话题最后回复时间更新

### Requirement: RealDailyQuestionService每日一问完整实现

系统 SHALL 实现每日一问的完整后端Real服务，替代Mock服务。

#### Scenario: 获取今日问题

* **WHEN** 用户打开每日一问页面

* **THEN** 系统查询当日DailyQuestion记录

* **AND** 展示问题内容和回答入口

* **AND** 若用户已回答，展示其回答和他人回答列表

#### Scenario: 提交回答

* **THEN** 系统创建DailyAnswer记录

### Requirement: RealCheckInService签到完整实现

系统 SHALL 实现签到的完整后端Real服务，替代Mock服务。

#### Scenario: 每日签到

* **WHEN** 用户点击签到按钮

* **THEN** 系统创建CheckIn记录（日期+用户ID）

* **AND** 计算连续签到天数

* **AND** 签到后获得额外推荐机会（+3次/天）

* **AND** 连续签到天数仅作视觉标识，无积分/等级

#### Scenario: 查询签到状态

* **WHEN** 用户打开签到页面

* **THEN** 系统返回今日是否已签到

* **AND** 返回连续签到天数

* **THEN** 系统记录点赞状态

* **AND** 帖子点赞数+1

#### Scenario: 用户重复点赞

* **WHEN** 用户再次点击已点赞的帖子

* **THEN** 系统取消点赞

* **AND** 帖子点赞数-1

* **AND** 不产生重复点赞记录

### Requirement: 前端Store Real模式全量验证

系统 SHALL 确保所有前端Store在Real模式下正常工作。

#### Scenario: Real模式全链路

* **WHEN** `VITE_API_MODE=real` 且后端运行在`db` Profile

* **THEN** 所有Store调用真实后端API

* **AND** 数据从数据库获取

* **AND** 错误处理正常（网络错误/业务错误/鉴权失败）

#### Scenario: profile.ts Mock模式支持

* **WHEN** `VITE_API_MODE=mock`

* **THEN** profile.ts使用本地Mock数据

* **AND** 无需后端服务即可运行前端

## MODIFIED Requirements

### Requirement: 推荐服务讨论推荐和历史记录（原为返回空列表）

RealRecommendationService SHALL 实现讨论推荐和推荐历史功能，替代空列表返回。

#### Scenario: 获取讨论推荐

* **WHEN** 用户访问讨论页面

* **THEN** 系统基于用户兴趣和热门话题返回讨论列表

* **AND** 不返回空列表

#### Scenario: 获取推荐历史

* **WHEN** 用户查看推荐历史

* **THEN** 系统返回用户的历史推荐记录

* **AND** 不返回空列表

### Requirement: 匹配算法优化（原为全量随机）

RealMatchService SHALL 使用智能匹配算法替代全量随机选择。

#### Scenario: 创建匹配

* **WHEN** 用户发起匹配请求

* **THEN** 系统基于推荐分数加权排序

* **AND** 使用分页查询替代findAll()

* **AND** 排除已喜欢/已有信号的用户

## REMOVED Requirements

### Requirement: Phase 1兼容方法硬编码用户ID

**Reason**: Real模式下应从JWT token获取当前用户ID，不应硬编码默认用户ID `1L`。
**Migration**: 所有Phase 1兼容方法改为从SecurityContext获取当前用户ID。

### 系统设计原则

1. **渐进式真实化**：Mock模式保留为开发回退，通过Profile切换mock/real
2. **模块独立**：每个Service独立可测，通过接口解耦
3. **数据共享**：利用现有User/Like/Post模型，新表通过外键关联
4. **双模式兼容**：大学生模式作为独立运营维度，通过campus字段贯穿全链路
5. **社交优先**：所有功能设计以互动为核心，延长停留时间

## 测试方法论

### 测试类型

| 类型             | 范围              | 工具                | 验收标准          |
| -------------- | --------------- | ----------------- | ------------- |
| 单元测试           | Service层业务逻辑    | JUnit + Mockito   | 覆盖率≥80%       |
| 集成测试           | API端到端          | Spring Boot Test  | 所有API 200响应   |
| 前端单元测试         | Store/Component | Vitest            | 覆盖率≥70%       |
| 契约测试           | 前后端数据模型         | OpenAPI Validator | Schema一致性100% |
| 回归测试           | 已有功能            | 手动+自动             | 零回归缺陷         |
| Mock/Real双模式测试 | 双模式切换           | 手动验证              | 两种模式均可用       |

### 关键验收标准

1. Real模式下兴趣圈完整可用（加入/退出/话题/回复）
2. Real模式下每日一问完整可用（获取问题/回答/查看他人回答）
3. Real模式下签到完整可用（签到/连续天数/额外推荐）
4. Real模式下临时聊天完整可用（创建会话/发消息/24h过期/联系方式交换）
5. 推荐算法包含兴趣标签匹配维度
6. heart\_signals表match\_type列存在且可用
7. 帖子点赞可取消，无重复点赞
8. 所有Store在Real模式下正常工作
9. profile.ts在Mock模式下正常工作
10. 大学生模式全链路生效
11. 无游戏化元素和购物功能
