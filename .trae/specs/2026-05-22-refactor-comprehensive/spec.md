# 恋爱小程序参考内容全面分析与重构方案 Spec

## Why

基于对参考目录（`D:\6\恋爱小程序\参考`）中11张竞品"青藤"微信截图的深度分析，结合对当前项目代码库的全面审计，发现项目虽经三期重构（`2026-05-21-reconstruct-social-app`、`2026-05-22-audit-gap-fill`、`2026-05-22-social-enhancement-phase3`），前端UI和Mock数据链路已基本完整，但存在以下核心问题：

1. **后端全量Mock，无真实数据持久化**：所有后端Service基于`MockRuntimeState`内存数据，无数据库连接、无JPA/MyBatis集成，服务重启后数据全部丢失
2. **核心社交功能后端为空壳**：VillageController/NotificationController/CircleController/DailyQuestionController等均为空壳，前端完整但后端无业务逻辑
3. **微信登录未对接**：AuthService忽略code参数直接返回Mock用户
4. **前后端数据模型不一致**：推荐卡片、帖子、签到等模型字段名和结构存在差异
5. **实时通信缺失**：聊天功能无WebSocket支持，消息无法实时推送
6. **推荐偏好设置前端入口存在但子包页面缺失**：`recommend-pref`分包已注册但页面未实现

本次方案的核心目标：**从"UI原型+Mock驱动"升级为"真实数据驱动+完整社交闭环"**，精准还原参考产品的核心功能体验，保留大学生模式的独特性，集成在线社交互动功能，确保推荐方案设置完整可用。

**特别约束**：严禁任何游戏化元素（积分/等级/排行榜/成就徽章等）和购物相关功能（商城/付费/虚拟物品等）。项目北极星指标：**提升用户引流效果 + 延长客户停留时间**。

### 二、交互流程分析

#### 核心用户旅程

```
微信登录 → 学校认证 → 资料完善 → 解锁全部功能
    ↓
寻觅页浏览推荐卡片 → 左滑拒绝/右滑喜欢
    ↓
双向喜欢 → 触发心动信号 → 倒计时内开聊
    ↓
私信对话 → 村口社区互动 → 活动报名参与
    ↓
每日推荐刷新 → 消息通知 → 社区动态更新 → 签到回流
```

#### 关键交互细节

1. **卡片滑动**：全屏沉浸式，手势流畅，飞出动画自然
2. **心动信号**：双向喜欢后自动触发，24h倒计时制造紧迫感
3. **村口发帖**：悬浮FAB按钮，随时可发，降低创作门槛
4. **私信入口**：多个触点（喜欢页/帖子详情/心动信号/兴趣圈），缩短社交距离
5. **时间门控**：每日中午12点刷新推荐，制造期待感和回流动力

## What Changes

### 一级：后端数据层基础设施（P0阻塞项）

- 配置Spring Boot数据源（MySQL + HikariCP）
- 集成JPA/Spring Data，创建Entity层对应14个Flyway迁移表
- 创建Repository层，替换`MockRuntimeState`
- 实现微信登录真实对接（code2Session + session_key解密）
- 统一前后端数据模型，消除字段名不一致

### 二级：核心社交功能后端实现（P1核心项）

#### 2.1 推荐引擎

- 实现`RecommendationService`真实推荐逻辑
- 基于校园（同校优先）、兴趣标签、课表空闲时间的加权推荐
- 每日限额+时间门控刷新逻辑
- 推荐偏好设置API（`GET/PUT /api/recommendations/preferences`）
- 推荐历史记录API

#### 2.2 村口社区

- 实现`VillageService`完整CRUD
- 帖子发布/详情/列表/删除
- 评论/点赞/转发逻辑
- 分类筛选和排序
- 同校优先排序

#### 2.3 喜欢与心动信号

- 实现`MatchService`喜欢/取消喜欢逻辑
- 双向喜欢检测→心动信号自动生成
- 访客记录逻辑
- 心动信号24h过期机制

#### 2.4 私信系统

- 实现私信会话创建/列表/消息发送
- 与临时聊天双轨并行
- 消息已读/未读状态
- 联系方式交换状态机

#### 2.5 实时通信

- 集成WebSocket（Spring WebSocket + STOMP）

- 在线状态展示

#### 2.6 通知系统

- 实现`NotificationService`通知生成逻辑

- 未读计数+标记已读

## 测试方法论

### 测试类型

| 类型 | 范围 | 工具 | 验收标准 |
|------|------|------|----------|
| 单元测试 | Service层业务逻辑 | JUnit + Mockito | 覆盖率≥80% |
| 集成测试 | API端到端 | Spring Boot Test | 所有API 200响应 |
| 前端单元测试 | Store/Component | Vitest | 覆盖率≥70% |
| 契约测试 | 前后端数据模型 | OpenAPI Validator | Schema一致性100% |
| 回归测试 | 已有功能 | 手动+自动 | 零回归缺陷 |
| 性能测试 | 推荐算法/消息推送 | JMeter | 推荐响应<500ms |

### 关键验收标准

1. 服务重启后所有数据保留（无内存依赖）
2. 微信登录真实可用（非Mock）
3. 推荐算法基于多维度加权（非硬编码）
4. 私信实时送达（WebSocket）
5. 互动通知即时生成
6. 兴趣圈完整链路可用
7. 每日一问完整链路可用
8. 推荐偏好设置影响推荐结果
9. 大学生模式全链路生效
10. 无游戏化元素和购物功能
