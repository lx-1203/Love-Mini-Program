# 恋爱小程序综合审查与社交功能增强方案 Spec

## Why

前两期重构（`2026-05-21-reconstruct-social-app` 和 `2026-05-22-audit-gap-fill`）已完成核心架构重构与差距填补，当前项目已具备：

* 五大一级入口（寻觅/喜欢/村口/消息/我的）完整实现

* 卡片式推荐发现 + 时间门控 + 回看

* 双向喜欢 → 心动信号 → 私信闭环

* UGC社区广场（三标签+六分类+发帖+互动）

* 资料完善硬门槛 + 完善度可视化

* 推荐计划偏好设置

* 线下活动（Mock数据驱动）

然而，通过对参考内容（`D:\6\恋爱小程序\参考` 中11张微信截图）的深度分析和当前代码库的全面审查，发现以下核心问题：

1. **线上社交互动深度不足**：当前社交链路偏"匹配+私信"单线模式，缺乏持续的在线互动场景来延长用户停留时间
2. **活动功能未真实化**：线下活动页面使用硬编码 Mock 数据，未对接后端 API，报名功能仅本地状态切换
3. **社区互动链路不完整**：村口页缺少"私信作者"快捷入口，帖子详情页缺少作者卡片交互
4. **用户留存机制薄弱**：除每日推荐时间门控外，缺乏日常打卡、互动提醒、动态推送等回流机制
5. **大学生模式完整性**：学校认证存在但缺少校园专属互动场景（如同校圈、校园话题等）

本次第三阶段的核心目标：在保留全部已有功能的基础上，**集成全新的在线社交互动功能**，以"提升用户引流效果"和"延长客户停留时间"为北极星指标。

**特别约束**：严禁任何游戏化元素（积分/等级/排行榜/成就徽章等）和购物相关功能（商城/付费/虚拟物品等）。

### 二、交互流程分析

参考产品的核心用户旅程：

1. **引流阶段**：微信登录 → 学校认证 → 资料完善 → 解锁全部功能
2. **发现阶段**：寻觅页浏览推荐卡片 → 左滑拒绝 / 右滑喜欢
3. **匹配阶段**：双向喜欢 → 触发心动信号 → 倒计时内开聊
4. **互动阶段**：私信对话 → 村口社区互动 → 活动报名参与
5. **留存阶段**：每日推荐刷新 → 消息通知 → 社区动态更新

## What Changes

### 一级：线下活动真实化（对接后端API）

* 活动 Store 从 Mock 数据切换为真实 API 调用（`GET /api/recommendations/activities`）

* 报名功能对接后端 API（`POST /api/recommendations/activities/{id}/enroll`）

* 活动详情页展示完整信息（活动描述、报名人数、参与者预览）

* 活动列表支持分页加载

### 二级：全新在线社交互动功能

#### 2.1 话题圈（Interest Circles）

* 基于兴趣标签的轻量级话题讨论组

* 用户可加入兴趣圈（如摄影圈、运动圈、读书圈、音乐圈）

* 圈内可发起话题讨论、分享动态

* 圈内成员列表展示、快速私信入口

* 与村口社区联动：圈子话题可精选到村口"兴趣圈"分类

#### 2.2 每日一问（Daily Question）

* 系统每日推送一个互动话题/问题

* 用户回答后可见他人回答

* 基于共同回答推荐匹配对象

* 增加日常活跃度和用户停留时间

* 回答可匿名或实名

#### 2.3 动态广场增强

* 村口帖子详情页增加作者交互卡片（展示学校/兴趣标签）

* 一键私信作者入口

* 帖子支持转发/分享（站内）

* 支持@提及用户功能

#### 2.4 互动提醒系统

* 被关注 / 被点赞 / 新评论 / 新访客 → 实时消息推送

* 每日推荐刷新提醒

* 心动信号即将过期提醒

* 兴趣圈新话题提醒

### 三级：用户留存机制

#### 3.1 日常签到

* 每日签到记录活跃状态

* 签到后获得额外推荐机会（+3次/天）

* 连续签到展示（非积分/等级，仅展示连续天数）

* 签到后解锁当日"每日一问"

#### 3.2 动态推送

* 系统消息推送：新推荐、新访客、新心动信号

* 社区动态推送：关注用户新发帖、兴趣圈新话题

### 四级：校园专属场景

#### 4.1 同校圈

* 基于学校的专属内容流

* 仅展示同校用户动态

* 同校推荐优先

* 校园话题标签（#食堂美食 #图书馆遇见 #校园活动）

#### 4.2 校园活动日历

* 与线下活动联动，展示校园活动日历视图

* 用户可标记"想参加"

* 同校用户可见参与意向

### 严格禁止的内容

* ❌ 游戏化元素：积分系统、等级体系、排行榜、成就徽章、虚拟货币、打卡奖励

* ❌ 购物相关：付费功能、虚拟物品购买、会员订阅、商品列表、交易功能

* ❌ 过度竞争：PK排名、人气榜、颜值评分

## ADDED Requirements

### Requirement: 线下活动真实化

系统 SHALL 将线下活动从 Mock 数据切换为真实后端 API 驱动。

#### Scenario: 活动列表加载

* **WHEN** 用户打开活动页

* **THEN** 系统调用 `GET /api/recommendations/activities` 获取活动列表

* **AND** 展示活动标题、地点、时间、报名人数

* **AND** 支持下拉刷新和分页加载

#### Scenario: 活动报名

* **WHEN** 用户点击"感兴趣"按钮

* **THEN** 系统调用 `POST /api/activities/{id}/enroll` 记录报名

* **AND** 按钮状态切换为"已感兴趣"

* **AND** 报名人数实时更新

#### Scenario: 活动详情

* **WHEN** 用户点击活动卡片

* **THEN** 展示完整活动信息（描述、时间、地点、参与者预览、报名人数）

### Requirement: 话题圈（Interest Circles）

系统 SHALL 提供基于兴趣标签的轻量级话题讨论组，增强在线社交互动。

#### Scenario: 发现和加入兴趣圈

* **WHEN** 用户在村口页或我的页访问"兴趣圈"

* **THEN** 展示兴趣圈列表：摄影圈、运动圈、读书圈、音乐圈、美食圈、旅行圈等

* **AND** 每个圈展示成员数量和最新话题摘要

* **AND** 用户可一键加入/退出

#### Scenario: 圈内话题讨论

* **WHEN** 用户进入兴趣圈

* **THEN** 展示该圈的话题列表：最新话题在前

* **AND** 用户可发起新话题（标题+正文+可选图片）

* **AND** 用户可回复他人话题

* **AND** 每个话题展示回复数和最后回复时间

#### Scenario: 从村口发现兴趣圈内容

* **WHEN** 用户在村口页选择"兴趣圈"分类

* **THEN** 展示来自各兴趣圈的精选话题

* **AND** 点击可进入该话题详情

### Requirement: 每日一问（Daily Question）

系统 SHALL 每日推送一个互动话题，用户回答后可查看他人回答，促进互动和发现。

#### Scenario: 回答每日一问

* **WHEN** 用户打开寻觅页或村口页

* **THEN** 展示"每日一问"卡片：问题标题+回答入口

* **AND** 用户可输入回答（限制200字）

* **AND** 提交后可见其他用户的回答列表

#### Scenario: 基于回答发现匹配

* **WHEN** 用户回答后浏览他人回答

* **THEN** 对于感兴趣的回答，可点击作者头像查看资料

* **AND** 可直接发起私信或右滑喜欢

* **AND** 共同回答可作为推荐算法的加分项

#### Scenario: 匿名回答

* **WHEN** 用户选择匿名回答

* **THEN** 回答显示"匿名用户"而非真实昵称

* **AND** 但仍可在后续选择解除匿名（仅对双向喜欢用户可见真实身份）

### Requirement: 互动提醒系统

系统 SHALL 提供实时互动提醒，增强用户回流的触发点。

#### Scenario: 收到新互动提醒

* **WHEN** 用户收到新的关注/点赞/评论/访客

* **THEN** 消息页系统通知中展示对应条目

* **AND** 消息TabBar显示未读红点

* **AND** 点击条目跳转到对应内容

#### Scenario: 心动信号过期提醒

* **WHEN** 心动信号倒计时剩余不足2小时

* **THEN** 系统推送提醒："心动信号即将过期，快开聊吧"

* **AND** 消息页Banner高亮提示

### Requirement: 同校圈

系统 SHALL 提供基于学校的专属内容流，强化大学生模式的独特性。

#### Scenario: 浏览同校动态

* **WHEN** 用户选择村口"同城"标签且已认证学校

* **THEN** 优先展示同校用户发布的帖子

* **AND** 帖子展示学校标签（如"校友"）

* **AND** 同校推荐排序优先

#### Scenario: 校园活动日历

* **WHEN** 用户从活动页切换到"校园日历"视图

* **THEN** 以日历形式展示当月校园活动

* **AND** 标注活动日期和标题

* **AND** 用户可标记"想参加"，同校用户可见参与意向

### Requirement: 日常签到

系统 SHALL 提供每日签到功能，作为活跃度标识和额外推荐机会的获取方式（非积分系统）。

#### Scenario: 每日签到

* **WHEN** 用户打开小程序

* **THEN** 若今日未签到，展示签到入口

* **AND** 签到后获得额外3次推荐机会

* **AND** 连续签到天数展示在日期旁（仅视觉标识，无等级/积分）

#### Scenario: 签到解锁每日一问

* **WHEN** 用户完成签到

* **THEN** 解锁当日"每日一问"回答权限

* **AND** 展示"每日一问"入口

## 技术架构建议

### 技术栈（沿用现有）

* **前端**：uni-app (Vue 3 + Pinia + TypeScript)

* <br />

* **后端**：Spring Boot (Java) + Maven

* **数据库**：MySQL + Flyway 迁移

* **API契约**：OpenAPI 3.0 (YAML)

* **测试**：Vitest (前端) / JUnit (后端)

### 系统设计原则

1. **渐进增强**：新功能叠加在现有模块上，不破坏已有功能
2. **模块独立**：话题圈、每日一问、同校圈各自独立为子包模块
3. **数据共享**：利用现有 User/Like/Post 模型，新表通过外键关联
4. **扩展预留**：为未来可能的语音/视频互动预留接口定义

### 新增数据表规划

* `interest_circles`：兴趣圈（名称、图标、成员数）

* `circle_memberships`：用户-兴趣圈关联

* `circle_topics`：圈内话题（标题、内容、图片、回复数）

* `circle_replies`：话题回复

* `daily_questions`：每日一问（日期、问题内容）

* `daily_answers`：用户回答（内容、是否匿名）

* `check_ins`：签到记录（日期、用户ID）

* `post_shares`：帖子转发记录
